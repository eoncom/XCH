## 1. Contexte & objectif

XCH n'a jamais fait l'objet d'un audit a11y automatisé (WCAG 2.0 AA). Avant
cutover pilote air-gap RSI (terrain), la baseline doit fixer :
- **Score Lighthouse a11y** par page (cible >= 70)
- **0 violation axe-core** `critical`/`serious` sur 8 pages standard
- **Justification documentée** pour 2 pages Konva (floor-plans, monitoring)

Cette mesure **n'est PAS bloquante E.4** sauf cas extrêmes (cf. §7 vigilance V2).
Elle alimente Track F (compliance suite) si le verdict global > seuils warning.

## 6. Konva exclusions (Décision D7)

**Pages concernées** :
- `/dashboard/floor-plans` — utilise `react-konva` ([FloorPlanViewer.tsx](../../frontend/src/components/floor-plans/FloorPlanViewer.tsx))
- `/dashboard/monitoring` — utilise `react-konva` (heatmap + topology canvas)

**Justification** : Konva rend HTML5 `<canvas>` natif. Les éléments dessinés
(`Stage`, `Layer`, `Image`, `Circle`, `Rect`) n'exposent pas d'ARIA exploitable
par axe-core / Lighthouse. Les violations détectées sont structurelles
(`image-alt` sur canvas non-applicable, `aria-required-attr` sur primitives
non-DOM) et non-actionnables sans re-architecturer la couche rendering.

**Mitigation** : ces 2 pages sont :
1. **scannées** (logs Playwright stdout + Lighthouse warn)
2. **non-bloquantes** dans le CI
3. **documentées explicitement** dans ce rapport
4. **éligibles Track F** si un client a11y-strict (RGAA / Section 508) est ciblé
   post-pilote → réécriture canvas → DOM SVG (perte performance acceptée)

## 7. Vigilance V2 — fallback policy

Selon les scores Lighthouse a11y observés post-run :

| Verdict | Action | Bloquant E.4 ? |
|---|---|---|
| 8/8 pages standard ≥ 0.70 | ✅ Baseline acceptée, archive | NON |
| 1-2 pages standard 0.50-0.70 | 🟡 Hotfix mineur PR2 (labels, contraste WCAG AA) | NON |
| ≥ 3 pages standard < 0.70 | 🟡 **V2 fallback** — créer [docs/perf/a11y-followup-track-f.md](a11y-followup-track-f.md), défer | **NON** |
| 1+ page standard < 0.50 | 🔴 **CRITICAL hotfix PR2 inclus** (R4.2) | 🟡 hotfix d'abord |

**Règle figée** : `< 70 sur >= 3 pages = défer Track F, PAS blocking E.4`.

## 8. Hotfix CRITICAL (vigilance R4.2)

Si verdict CRITICAL (page < 50) → PR hotfix dans PR2 :
- **Labels manquants** : `<label htmlFor>` ou `aria-label` sur inputs / buttons sans texte visible
- **Contraste WCAG AA** : ratios > 4.5:1 (texte normal) / 3:1 (large 18pt+) — vérifier
  Tailwind HSL CSS vars `--foreground` / `--muted-foreground` / `--primary-foreground`
- **Focus order** : `tabindex` cohérent, pas de `tabindex > 0`
- **Landmark roles** : `<main>`, `<nav>`, `<header>`, `<footer>` présents

Re-run workflow `a11y.yml` post-hotfix, comparaison avant/après dans §4/§5.

## 9. Procédure de run (canonique)

### 9.1 CI (recommandé)

Automatique :
- Push sur `main` ou `develop` → workflow run
- PR modifiant `frontend/**` → workflow run

Manuel :
```bash
gh workflow run a11y.yml
gh run list --workflow=a11y.yml --limit 1
gh run download <run-id>
```

### 9.2 Local

Pré-requis : backend + frontend lancés (idéalement via docker-compose.ci.yml pour
single-origin).

```bash
# Stack single-origin (recommandé)
docker compose -f docker-compose.ci.yml up -d --build
curl -X POST http://localhost:8080/api/setup/initialize -H 'Content-Type: application/json' \
  -d '{"organizationName":"local","subdomain":"local","adminName":"a","adminEmail":"admin@demo.fr","adminPassword":"Demo1234","loadDemoData":true}'

cd frontend
PLAYWRIGHT_BASE_URL=http://localhost:8080 PLAYWRIGHT_API_URL=http://localhost:8080 \
  npm run test:a11y

LHCI_BASE_URL=http://localhost:8080 npm run lhci:autorun
```
