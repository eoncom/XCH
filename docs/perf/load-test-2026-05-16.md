## 1. Contexte & objectif

XCH n'a jamais été soumis à charge réaliste (≤ pilote air-gap 1 client RSI). Le
risque non-mesuré au moment d'E.4 PR2 est :
- N+1 latents sur `/api/assets` (10k+ rows réels attendus pilote)
- Performance index composite `audit_logs(tenantId, delegationId, timestamp)`
  ajouté PR1 — jamais stressé sur > 100 rows
- BullMQ queue depth sous load mixed read/write
- RSS worker peak (config dev mémoire suffisante ?)

Cette baseline mesure ces 4 axes avant pilote, et fixe les seuils SLA pour
Track G (perf consolidée) si dépassements détectés.

### 8.2 Local (dev workstation)

```bash

# 2. Boot backend infra + backend daemon
cd backend
docker compose up -d postgres redis minio minio-init
npx prisma migrate deploy
npm run build && nohup node dist/main.js > /tmp/backend.log 2>&1 &
# 3. Seed loadtest fixture
npx ts-node scripts/seed-loadtest.ts --reset
# 4. Run scenarios
cd ..
for s in smoke-load read-heavy write-mixed audit-heavy; do
  k6 run --summary-export=/tmp/k6-$s.json tests/load/scenarios/$s.js
done
# 5. Capture metrics §5 + analyse §6
# 6. Compléter §4 + §7 + commit
```

## 9. Hotfix policy (vigilance R3.4)

Si N+1 détecté sur endpoint smoke (`/api/assets`, `/api/sites`) OU p95 read >
1s : **PR hotfix inclus PR2** :
- Ajout `include: { site: true, delegation: true }` ou `select` ciblé
- Bump `BULL_WORKER_CONCURRENCY` (défaut 5 → 10) si queue depth > 100
- Index manquant détecté par EXPLAIN → ajout migration Prisma (backlog Track G
  acceptable si non-critique)

Re-run scénario post-fix, comparaison avant/après dans §4 (nouvelle colonne
"après hotfix").

## 10. Suite Track G (perf consolidée) — backlog

Items différés Track G :
- N+1 fix consolidé multi-endpoints (au-delà du smoke)
- BullMQ worker concurrency tuning + retry strategies
- Cache Redis `/api/assets`, `/api/sites` (5min TTL) si p95 > 300ms
- Index Prisma secondaires si EXPLAIN flag full table scan
- Connection pool tuning (`DATABASE_URL?connection_limit=20`) si Postgres
  connections > 80% pool
